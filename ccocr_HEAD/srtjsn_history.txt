srtjsn.py 開発履歴  260327  cy
========================================

【目的】
jsn4db の lines 配列を読み取り順（上から、段組は左段→右段）に並び替える。
words も lft 昇順で並び替える。

【テスト対象】
rot.png.STRAIGHT.png.DI.json（２段組の請求書）
- 左段（col 0）: COMMERCIAL INVOICE, Invoice No, Invoice Date, Currency,
               Seller, ABC Trading, 1-2-3 Example Street, Tokyo, Country of Origin
- 右段（col 1）: Buyer, XYZ Manufacturing, 456 Sample Ave, Los Angeles
- 下部（items table）: No, Description, Qty, Unit Price, Amount,
                     1, Product A, 2, ...

【正しい順番】
lines[0]  = COMMERCIAL INVOICE
lines[1]  = Invoice No
lines[2]  = Invoice Date
lines[3]  = Currency
lines[4]  = Seller
lines[5]  = ABC Trading
lines[6]  = 1-2-3 Example Street
lines[7]  = Tokyo
lines[8]  = Country of Origin
lines[9]  = Buyer         ← ここが問題箇所
lines[10] = XYZ Manufacturing
lines[11] = 456 Sample Ave
lines[12] = Los Angeles
lines[13以降] = items table (top 順)

========================================
トライ＆エラー履歴
========================================

【v1】greedy row merge（失敗）
------------------------------------------
問題:
  _group_rows で行を row に追加するとき、row の btm を全メンバーの max に
  拡張していたため、チェーンマージが発生。
  全 lines が1つの巨大な row に統合される → col=0 ばかり → セパレータ未検出。

結果: lines 全部 col 0 → top 順ソートだけになり段組未認識。

修正方針: 水平方向に重複しない場合のみ row に参加できる制約を追加。

【v2】水平非重複制約追加（部分的に改善・まだ失敗）
------------------------------------------
_can_join() を追加:
  - 垂直重複: max(top_a, top_b) < min(btm_a, btm_b)
  - 水平非重複: max(lft_a, lft_b) >= min(ryt_a, ryt_b)  (重複なし)
  両方満たす場合のみ row に参加。

問題:
  単一ラインの row（1メンバーのみ）も _find_separators に使っていたため、
  単一ラインの row は「自分以外の全 x 座標」がギャップ扱いになり、
  偽のセパレータが大量発生。

【v3】multi-line rows のみをセパレータ検出に使用（改善・まだ失敗）
------------------------------------------
_find_separators 内で:
  multi = [row for row in rows if len(row) >= 2]
  if len(multi) < 2: return []

問題:
  _MIN_COV = 0.8 が厳しすぎた。
  rot.png のケース: multi-line rows が 4 行（Seller/Buyer 段のペア×3 + items行ペア×1）
  x=297-517 のギャップが 3/4 行 = 75% < 80% → セパレータ未検出。

修正: _MIN_COV を 0.75 に下げる。

【v4】_MIN_COV=0.75（セパレータ検出OK・並び順まだ失敗）
------------------------------------------
セパレータは正しく検出されるようになった（x≈350 付近）。

問題:
  items table の "No"（lft=176, center=191, top=715）が col 0 に分類される。
  Buyer（col 1, top=212）より top が大きいが、col が小さいため:
    sort key: (col=0, top=715) < (col=1, top=212)
  → "No" が "Buyer" より前に出力されてしまう。

具体的症状:
  lines[8]  = Country of Origin  (正)
  lines[9]  = No                 (誤 → Buyer であるべき)
  lines[10] = Buyer              (誤)

【v5】_COL_BUFFER 導入（実装済み・未テスト）
------------------------------------------
考え方:
  段組ヘッダ部分（multi-line rows が存在する領域）だけをカラムソートの対象にする。
  items table などはカラムソートから除外し、top 順で後ろに追加する。

実装:
  _COL_BUFFER = 100  # normalized coords

  multi = [row for row in rows if len(row) >= 2]
  maxbtm = max(max(m['btm'] for m in row) for row in multi)
  colmax = maxbtm + _COL_BUFFER

  col_lines  = [ln for ln in lines if ln['top'] <= colmax]   # カラムソート対象
  post_lines = [ln for ln in lines if ln['top'] >  colmax]   # top 順で後追加

  tagged = sorted(
      ((_col_of(ln, seps), ln) for ln in col_lines),
      key=lambda t: (t[0], t[1]['top']))
  result = [ln for _, ln in tagged] + sorted(post_lines, key=lambda l: l['top'])

rot.png での動作:
  last_multi_btm ≈ 368
  colmax = 368 + 100 = 468
  col_lines:  top ≤ 468 → 段組ヘッダ全部（Seller 段 + Buyer 段）
  post_lines: top > 468 → items table（No, 1, 2, ...）

期待結果:
  col_lines の col 0 (top 順) → COMMERCIAL INVOICE ... Country of Origin
  col_lines の col 1 (top 順) → Buyer, XYZ Manufacturing, 456 Sample Ave, Los Angeles
  post_lines (top 順)         → No, Description, 1, Product A, ...
